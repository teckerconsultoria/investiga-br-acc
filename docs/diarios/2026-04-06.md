# Diário Técnico — 2026-04-06

## Objetivo da Sessão

Implementar scripts de download para três fontes ETL que tinham pipeline mas nunca tiveram coleta automatizada (TCU, TransfereGov, PGFN), executar os downloads e ETLs na VPS, e produzir um plano de verificação para as demais fontes do projeto.

## O que foi Feito

- Criados três novos scripts de download: `download_tcu.py` (4 CSVs diretos do TCU), `download_transferegov.py` (ZIP único do Portal da Transparência), `download_pgfn.py` (ZIP trimestral de dívida ativa da União)
- TCU e TransfereGov executados com sucesso na VPS: 45.369 sanctions e 71.871 amendments carregados no Neo4j
- PGFN corrigido para processamento em streaming após OOM crash (24M registros → pipeline reformulado para flush por chunk)
- VPS estabilizada: swap de 4GB adicionado, Docker restaurado após OOM
- Plano de verificação `docs/verification_plan_downloads_etl.md` criado e revisado: 9 fontes verificadas, 23 no Tier 1 de diagnóstico, fontes GCP descartadas

## Decisões Técnicas

- **Streaming no pipeline PGFN**: o padrão extract→transform→load acumulava 24M registros em duas listas Python (~2GB RAM), causando OOM no servidor com 3.8GB e sem swap. A solução foi mover o transform inline no `load()`, fazendo flush de cada chunk de 100K linhas diretamente para o Neo4j antes de ler o próximo. O `seen_inscricoes` (set de deduplicação) ainda cresce em memória, mas é aceitável (~400MB no pior caso).
- **Swap no VPS**: a VPS Hostinger não tinha swap configurado. Com cargas ETL pesadas, qualquer pipeline com uso de memória acima de 3GB causaria OOM kill imediato. Swap de 4GB adicionado como medida permanente (`/etc/fstab`).
- **Descarte de fontes GCP do plano**: `caged`, `cnpj_bq`, `dou`, `mides` e `stf` removidos do escopo ativo — todos dependem de BigQuery com billing project externo, tornando a verificação inviável sem credenciais GCP dedicadas.
- **Plano em modo diagnóstico**: em vez de re-executar downloads de todas as fontes, o plano foi reorientado para auditoria estática (URLs, campos esperados vs. schema atual) — menos destrutivo e mais rápido de executar.

## Bugs Corrigidos

- **OOM no pipeline PGFN**: acúmulo de 24M registros em memória antes do load → solução: flush por chunk no Neo4j durante o próprio loop de transformação
- **`rows_in = 0` em TCU/TransfereGov/PGFN**: pipelines novos não setavam `self.rows_in` antes do check do base class, causando `RuntimeError: No data found` mesmo com dados carregados → corrigido adicionando `self.rows_in = len(...)` no início de `load()`
- **Quebra de linha no paste SSH**: comandos multi-linha colados no terminal da VPS eram interpretados como dois comandos separados, quebrando redirects e argumentos → solução: gerar script via SSH e executar o arquivo
